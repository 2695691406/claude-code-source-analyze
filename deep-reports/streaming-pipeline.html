⚡ 流式处理与并发架构

本报告聚焦 Claude Code 在查询管道、工具执行器、消息队列与取消控制中的流式设计。分析严格建立在 restored-src/src/query.ts、services/tools/StreamingToolExecutor.ts、 services/tools/toolOrchestration.ts、utils/generators.ts 与 utils/messageQueueManager.ts 等文件的可见事实之上，重点解释它如何用 AsyncGenerator 把 UI 流、工具流、取消流与回退流整合成一个可中断、可限流、可恢复的架构。

🏗️ 架构总览

Claude Code 的流式处理不是把“模型响应流”单独做成一个组件，而是让查询入口、工具执行、消息更新和取消信号都服从同一种生成器范式。核心意义在于：每一层都能边生产、边消费、边中断，因此上层 UI 不必等待整轮工具调用完成，底层执行器也不必把所有结果缓存在一次性结构里。

从查询到工具结果的统一数据流

┌────────────────────────────────────────────────────────────────────┐
│ query() / queryLoop() │
│ yield StreamEvent / RequestStartEvent / Message ... │
├────────────────────────────────────────────────────────────────────┤
│ StreamingToolExecutor 或 runTools() │
│ ├─ 并发安全工具：批量并行，受并发上限约束 │
│ ├─ 非并发工具：逐个执行，避免写入冲突 │
│ └─ 按消息增量回流 MessageUpdate │
├────────────────────────────────────────────────────────────────────┤
│ messageQueueManager │
│ now > next > later，供 UI 与命令处理层按优先级消费 │
└────────────────────────────────────────────────────────────────────┘

架构关键词

流式事件 并发分区 取消传播 应用层背压

这里的“背压”更接近应用层流量治理：通过 CLAUDE_CODE_MAX_TOOL_USE_CONCURRENCY、生成器排空策略、消息优先级和 fallback/discard 机制，避免工具风暴或中断场景把 UI 与执行层拖入失控状态。

分析边界

本文只解释代码中直接可见的结构：类型签名、状态机、函数边界、消息优先级与取消链。不会虚构底层线程池、网络层协议细节或不存在的调度器实现。

🔄 AsyncGenerator 主干

这一架构最关键的事实，是 query() 自身就不是一次性返回值，而是查询全生命周期的异步生成器。它把“模型流事件”、“请求开始信号”、“消息对象”、“墓碑消息”和“工具摘要消息”放进同一条可增量消费的输出流里。

query() 的精确返回类型

export async function* query(
  params: QueryParams,
): AsyncGenerator<
  StreamEvent
  | RequestStartEvent
  | Message
  | TombstoneMessage
  | ToolUseSummaryMessage,
  Terminal
>

这一定义非常重要：yield 的是运行中事件流，return 的却是 Terminal。也就是说，调用方通过 for await...of 消费过程，而“终止状态”被保留为生成器完成值，不和中途消息混在一起。

为什么说 AsyncGenerator 是“贯穿式模式”

层级 代码事实 架构含义
查询入口 query() 返回复合类型的 AsyncGenerator<..., Terminal> 整个会话轮次天然支持增量输出与中途终止
工具执行 runTools() 与 getRemainingResults() 也都以生成器输出更新 工具层无需等待“所有工具全部完成”才向上游汇报
工具函数 generators.ts 提供 all()、returnValue()、fromArray() 等通用适配器 说明生成器不是点状使用，而是被提炼成基础编排原语
关键解读： Claude Code 的查询循环不是“拿到完整 assistant message 后再执行工具”，而是先把模型流、工具执行和结果补全纳入同一条异步可消费管道。这个选择使流式 UI 与工具回流天然兼容。

⚙️ StreamingToolExecutor

StreamingToolExecutor 是流式工具执行的核心。它并不只负责“执行工具”，还负责把工具状态机、进度消息、已完成结果与等待中结果拆分为两类生成器视图，因此可以在查询主循环里持续轮询已完成更新，同时保留一个“收尾排空”路径。

双生成器结果流 + 状态机

type ToolStatus = 'queued' | 'executing' | 'completed' | 'yielded'

class StreamingToolExecutor {
  getCompletedResults(): Generator<MessageUpdate>
  async *getRemainingResults(): AsyncGenerator<MessageUpdate, void>
}

状态流： 'queued' → 'executing' → 'completed' → 'yielded'

getCompletedResults()

这是同步生成器，职责是把已经完成、且尚未上抛的结果按顺序吐给上层。它更像一种“非阻塞轮询窗口”：如果某些工具已经结束，就立即把 MessageUpdate 交出去；如果没有，就什么也不阻塞地返回。

getRemainingResults()

这是异步生成器，职责是等待所有仍未收集的结果完成并逐个回流。它用于查询循环收尾、取消后排空、以及 fallback 之后补全缺失的 tool_result 块。

为什么是“双通道”而不是单通道

1
运行中轮询
主循环可以反复调用 getCompletedResults()，尽快把已完成工具结果送回 UI。
2
结束时排空
在正常收尾、用户中断或 streaming fallback 后，再用 getRemainingResults() 兜底等待所有必要结果。
3
状态闭环
只有当结果真正被向上层交付后，状态才进入 'yielded'，这保证了“完成”与“已消费”不是同一个概念。

架构图：工具状态与消息回流

┌─────────┐ start ┌────────────┐ result ready ┌────────────┐
│ queued ├────────────────▶│ executing ├──────────────────────▶│ completed │
└─────────┘ └────────────┘ └─────┬──────┘
│ pendingProgress 可先行 yield │
└───────────────────────────────────────────────────────────────────┘
▼
┌─────────┐
│ yielded │
└─────────┘

🧩 runTools() 与并发分区

Claude Code 并不是“所有工具一股脑并行”。toolOrchestration.ts 中的 runTools() 先经过 partitionToolCalls()，把连续的工具调用切成“可安全并发批次”和“必须单独执行的批次”。这说明并发策略来自工具语义，而不是统一线程数配置。

核心编排接口

export async function* runTools(
  toolUseMessages: ToolUseBlock[],
  assistantMessages: AssistantMessage[],
  canUseTool: CanUseToolFn,
  toolUseContext: ToolUseContext,
): AsyncGenerator<MessageUpdate, void>

runTools() 是“无 streaming executor 时”的直接执行入口，但它仍然保留了生成器输出形式，因此与整体流式架构兼容。

partitionToolCalls() 的语义

分区类型 判定依据 执行方式 含义
并发安全批次 工具被标记为 concurrency-safe 同一批内并行执行 适合读取型、互不修改共享状态的调用
非并发批次 工具不是 concurrency-safe 单个批次只放一个工具，串行执行 避免 Bash / 编辑 / 写操作相互踩踏

并发上限：默认 10

function getMaxToolUseConcurrency(): number {
  return (
    parseInt(process.env.CLAUDE_CODE_MAX_TOOL_USE_CONCURRENCY || '', 10) || 10
  )
}

这不是无限并发，也不是写死在线程池里的系统参数，而是一个清晰暴露给运行环境的应用层限流阈值，默认值为 10。

为什么这就是背压的一部分

如果模型一次性产出大量可并发工具调用，partitionToolCalls() 负责先做语义切分，getMaxToolUseConcurrency() 再限制同时在飞数量。这样可以防止工具执行层瞬时爆炸，也避免 UI 端同时面对过量结果流入。

关键事实： 这里的工具编排明确区分 runTools() 与 partitionToolCalls()。前者是执行入口，后者是安全边界；“并发”并不是默认权利，而是只有被判定为 concurrent-safe 的工具才拥有。

🛠️ generators.ts 工具链

Claude Code 没有把生成器当成语法糖使用，而是专门抽出 utils/generators.ts 作为编排原语库。这里的辅助函数解释了为什么上层能轻量拼装并发与流式行为，而不需要每个模块都自己重新写 drain / gather / convert 逻辑。

四个关键辅助函数

函数 签名/角色 架构作用
all<A>() 并发消费多个 AsyncGenerator，支持 concurrencyCap 为工具并发提供统一“边跑边产出”的合流器
lastX() 消耗完整个异步生成器并返回最后一个 yield 值 当中间值无关、只关心最终一项时可直接收口
returnValue() 排空生成器后提取其 return 值 把“过程输出”和“最终返回值”分开处理
fromArray<T>() 把数组适配成异步生成器 让非流式源也能接进统一生成器管道

all<A>() 的架构意义

export async function* all<A>(
  generators: AsyncGenerator<A, void>[],
  concurrencyCap = Infinity,
): AsyncGenerator<A, void>

这不是简单的 Promise.all() 替代品。它保留了“多路输入、单路增量输出”的形态，并把并发上限直接做进原语层。runToolsConcurrently() 使用它，说明 Claude Code 把并发工具执行建模为“多个会不断 yield 的源”，而不是“多个只会 resolve 一次的 promise”。

returnValue() 与 lastX()

这两个函数共同强调了生成器的双重语义：一方面它在运行中持续产出值，另一方面它在结束时还可以返回独立的完成值。query() 返回 Terminal，正是这种分离语义的上层体现。

fromArray<T>()

这个辅助函数看似简单，但意义很大：它让非流式数据源也能伪装成异步流，从而和流式工具结果、消息流、收尾流程共享同一套消费协议。

📬 messageQueueManager.ts

除了模型/工具流本身，Claude Code 还维护一个命令消息队列，用于协调用户输入、通知和系统消息。这个模块值得注意的地方在于：它同时面向 React 订阅模型和非 React 读取模型，因此既有快照语义，也有优先级语义。

优先级定义

const PRIORITY_ORDER: Record<QueuePriority, number> = {
  now: 0,
  next: 1,
  later: 2,
}

这三档优先级说明命令队列不是 FIFO 的纯日志，而是可以把紧急动作、普通输入和延后通知分层消费。它和工具并发限流一起，构成 UI 侧的应用层背压治理。

React 集成：useSyncExternalStore

模块导出 subscribeToCommandQueue 与 getCommandQueueSnapshot()，正适合 React 通过 useSyncExternalStore 订阅外部状态。这里的关键不是 Hook 名字本身，而是它采用“可订阅 + 不可变快照”模式，让界面层能可靠感知队列变更。

非 React 消费

同一模块又保留了直接读取和出队接口，说明这套队列不是纯 UI store，而是跨终端界面、打印层和命令处理层共享的基础设施。

优先级 + 快照 = 可控的 UI 流

1
写入时定优先级
now、next、later 把消息分成不同紧迫等级。
2
读取时按优先级出队
消费端不必先扫完整条队列，最高优先级项先被处理。
3
React 用快照订阅
useSyncExternalStore 风格接入可减少竞态，保证渲染看到一致状态。

🛑 三层取消链

Claude Code 的取消模型并非单一 AbortController。在工具执行流中，至少可以观察到三层结构：根控制器、兄弟级控制器和每个工具自己的控制器。它们的关系决定了“用户中断”、“某个工具失败”与“某个工具自行 abort”分别传播到哪里。

三层层级结构

root toolUseContext.abortController
  └─ layer 2 siblingAbortController
      └─ layer 3 toolAbortController (per tool)

Root：全局工具轮次取消

toolUseContext.abortController 代表整个工具使用上下文的根中断信号。用户中断、上层查询退出等事件最终都会体现在这一层上。

Sibling：同批工具的联动取消

siblingAbortController 作为根的子控制器，主要用于在一个工具出错后终止同批次或相关兄弟工具，而不是立即等同于整个会话根取消。

Per-tool 控制器的意义

每个工具还会创建自己的 toolAbortController。这样做的效果是：单个工具可以拥有本地 abort 生命周期，同时又能在需要时把原因上抛；但某些特殊原因（例如兄弟错误）不会无条件冒泡成根级取消，避免一次局部联动把整轮状态错误放大。

架构图：取消传播方向

用户中断 / 查询结束
│
▼
toolUseContext.abortController
│ 向下传播
▼
siblingAbortController
│ 再向下传播
▼

│
├─ 普通原因：可向上冒泡到 root
└─ sibling_error：用于兄弟联动，不等同于全局失败

🚨 回退、丢弃与背压

真正体现这套架构成熟度的，不是“成功路径”，而是它如何处理 Bash 错误、用户中断，以及 streaming fallback 后的丢弃状态。代码中的 discarded 概念与 synthetic error message 生成逻辑，说明系统非常在意保持 tool_use / tool_result 对齐，不让失败尝试污染后续轮次。

Bash 错误的兄弟中止语义

在 StreamingToolExecutor 中，只有 Bash 错误会触发 siblingAbortController.abort('sibling_error')。这一点非常有辨识度：并不是任何工具失败都连坐取消，而是只有像 Bash 这类天然存在命令链依赖的工具，其失败才被视为足以让兄弟工具失去继续执行意义。

Bash error → sibling_error 独立读取工具失败 ≠ 全部取消

用户中断处理

当根 abort reason 为 'interrupt' 时，执行器会根据工具自身的 interrupt behavior 决定是返回 'user_interrupted' 合成错误，还是允许某些工具保持 block 行为。说明“按 ESC”并不是粗暴终止一切，而是尊重工具类型差异。

Streaming fallback / discarded

当流式尝试被放弃时，执行器会进入 discarded 状态，并把后续中止原因解释为 'streaming_fallback'。其目的不是隐藏错误，而是生成合成结果，避免旧尝试里的工具结果悬空或错配到新一轮流中。

失败路径如何保持消息一致性

正常路径：
tool_use → executeTool → tool_result

用户中断：
tool_use → root abort('interrupt') → synthetic tool_result ('user_interrupted')

Bash 兄弟失败：
bash tool error → sibling abort('sibling_error') → 兄弟工具生成取消结果

streaming fallback：
discard() → discarded = true → synthetic tool_result ('streaming_fallback')

把这些事实放回整体架构中的解释

  • 并发上限 10 控制的是工具风暴的入口压力。
  • partitionToolCalls() 防止语义不安全的工具并行。
  • getCompletedResults() / getRemainingResults() 保证运行中可增量、结束时可排空。
  • messageQueueManager 的 now / next / later 让 UI 消费优先级化。
  • 三层 AbortController 让取消既可联动，又不至于无限扩大。
  • discarded 与 synthetic result 则保证失败路径依旧满足消息配对约束。
最终结论： Claude Code 的流式与并发架构，本质上是一个“生成器优先”的应用层控制面。AsyncGenerator 负责统一增量输出协议，runTools() 与 StreamingToolExecutor 负责把工具执行转成可限流的流，消息队列负责优先级消费，而三层取消链与 discarded/fallback 机制则负责把失败路径也收束到一致的消息模型内。
