状态总览：从共享上下文到可恢复计量

Claude Code 的状态管理不是单点 store，而是按职责拆开的多层组合：UI 交互与渲染局部状态通过 Context 分发；消息和异步事件通过队列驱动；长期配置通过迁移脚本维持兼容；运行期的成本、时延与代码变更量则由成本追踪模块记录。这种结构使它既能支撑终端 UI 的即时反馈，也能支撑会话恢复与配置升级。

架构骨架

输入/异步事件
    │
    ├─► Mailbox / Queue
    │       │
    │       └─► React 组件树消费消息
    │
    ├─► Context Providers
    │       ├─ QueuedMessage / Mailbox / Modal / Overlay
    │       ├─ Notifications / PromptOverlay / Voice
    │       └─ FpsMetrics / Stats
    │
    ├─► 迁移层（main.tsx 启动时执行）
    │       └─ 旧设置 → 当前 settings 结构
    │
    └─► cost-tracker.ts
            ├─ API 成本与耗时
            ├─ Tool 耗时
            ├─ 代码增删行
            └─ modelUsage 会话统计

上下文层负责“共享当前状态”

九个 Context 文件把消息排队、模态框、悬浮层、通知、语音、性能采样等状态拆开管理，避免把所有内容压进单一全局对象。这样每一类状态都有清晰的读写边界与消费面。

迁移层负责“保住历史兼容”

11 个迁移脚本说明设置模型并非静态设计，而是在版本演进中不断整理字段与默认值。迁移层让旧配置仍能在新版本中落到当前结构，而不是把兼容逻辑散落到 UI 与业务代码里。

9 个 Context 层：按职责拆分的共享状态面

报告中的“9 个 Context 文件”如下，全部来自 src/context 目录，并构成 Claude Code 在 React 树内共享状态的主要横切层：

文件 关注点 架构含义
QueuedMessageContext.tsx 排队消息的布局上下文 说明消息不是“来一条画一条”，而是存在带顺序语义的 UI 排队层。
fpsMetrics.tsx FPS/渲染度量入口 将性能采样纳入 UI 状态体系，而非完全外置。
mailbox.tsx 消息队列注入 把异步消息收发抽象为共享邮箱，是队列驱动 UI 的关键接点。
modalContext.tsx 模态框尺寸/滚动上下文 表明终端 UI 中的模态层拥有独立布局坐标系。
notifications.tsx 通知队列 即时提示不是直接副作用，而是被排入统一通知流。
overlayContext.tsx 覆盖层活跃态 用于协调 escape、焦点和遮罩优先级等跨组件交互。
promptOverlayContext.tsx 输入提示覆盖层 把建议、选中项、对话框状态从输入框本体中解耦出来。
stats.tsx 统计 store / 指标聚合 说明可观测性是 React 状态的一部分，而不是事后日志。
voice.tsx 语音状态 语音录制、处理中间态、错误态被显式建模并向组件树共享。

为什么要拆成九层而不是一个“大状态对象”

从这组文件名可以看出，Claude Code 把状态按交互域切开：消息域、覆盖层域、提示域、通知域、语音域、性能域。这样的拆分能减少无关组件重渲染，也让不同 UI 子系统以更明确的 API 暴露读写能力。

队列驱动消息流：终端 UI 不是同步直写，而是消息编排

Claude Code 的一个核心架构特征是队列驱动。就名字与接线方式而言，mailbox.tsx 和 QueuedMessageContext.tsx 说明消息流先进入队列/邮箱，再由 UI 层按顺序消费；notifications.tsx 则把瞬时提示也纳入队列语义，而非让任何组件直接向屏幕“抢写”。

消息流的架构级解释

用户输入 / 系统事件 / 任务结果
            │
            ▼
      Mailbox / Queue
            │
   ┌────────┼────────┐
   ▼        ▼        ▼
主消息区   通知流   覆盖层/提示层
   │        │        │
   └────────┴────────┘
            ▼
      React 渲染提交

队列化的直接收益

  • 异步来源可以被统一消费，而不是每个来源单独驱动 UI。
  • 顺序语义更清楚：尤其适合对话、通知、排队消息。
  • UI 可以在消费消息前后附着布局、折叠、优先级等策略。

与 Context 的关系

Mailbox 负责“消息进出”，QueuedMessage/Notifications/Overlay 等 Context 负责“这些消息在 UI 里怎么被共享与解释”。因此它不是二选一，而是 队列负责流动，Context 负责传播。

架构含义归纳
- mailbox.tsx               = 异步消息入口 / 共享邮箱
- QueuedMessageContext.tsx  = 排队消息的显示语义
- notifications.tsx         = 提示流的排队与展示
- overlayContext.tsx        = 队列消费后的交互覆盖层协调

这类设计特别适合终端环境。终端不像浏览器那样拥有天然的多层窗口系统，因此应用内部必须自己建模消息、覆盖层与优先级。Claude Code 通过队列与 Context 的组合，为终端 UI 提供了一个接近事件总线 + 声明式渲染的中间层。

设置迁移演进：11 个脚本维持状态模型的向后兼容

Claude Code 的设置状态并不是一次性定型，而是通过迁移脚本持续演化。以下 11 个文件完整构成此次报告要求覆盖的迁移清单：

# 迁移文件 演进信号
1migrateAutoUpdatesToSettings.ts自动更新相关配置从旧位置迁移到 settings 体系。
2migrateBypassPermissionsAcceptedToSettings.ts危险权限接受状态被整理到设置层。
3migrateEnableAllProjectMcpServersToSettings.ts项目级 MCP 服务器启用策略迁入 settings。
4migrateFennecToOpus.ts模型命名/默认值从旧别名迁向当前体系。
5migrateLegacyOpusToCurrent.tsLegacy Opus 规格被统一到当前模型表示。
6migrateOpusToOpus1m.tsOpus 默认档位继续向新表示演进。
7migrateReplBridgeEnabledToRemoteControlAtStartup.ts远程控制/桥接启动字段重命名并迁移。
8migrateSonnet1mToSonnet45.tsSonnet 型号版本被提升到更具体的新代号。
9migrateSonnet45ToSonnet46.ts继续从 4.5 演进到 4.6 语义。
10resetAutoModeOptInForDefaultOffer.ts自动模式默认 offer 发生变化，需要重置 opt-in 状态。
11resetProToOpusDefault.tsPro 用户的默认模型选择被重置到新的默认值。

迁移层的架构意义

这些文件共同表明：Claude Code 把“设置演进”当作独立系统，而不是在运行时遍布 if/else 去兼容历史字段。模型命名、权限确认、MCP 开关、自动模式默认值，都通过迁移层收敛到当前设置结构中。

启动阶段的状态演化
旧配置 / 旧字段 / 旧模型名
             │
             ▼
       migration runner
             │
             ├─ 字段重命名
             ├─ 默认值重置
             ├─ 模型别名升级
             └─ 设置位置迁移
             ▼
      当前 settings 结构

对数据流而言，迁移层的作用是“在 React 树启动前清理状态输入”。只有把历史配置折叠到当前结构，后面的 Context 与 UI 才能假设自己读取的是一致语义，而不是多个年代的混合格式。

成本与指标追踪：cost-tracker.ts 是运行态会计账本

cost-tracker.ts 暴露了一组非常关键的状态字段，它们把 Claude Code 的 API 调用、工具耗时以及代码改动量转成可恢复、可展示、可聚合的会话指标：

字段 含义 架构解读
totalCostUSD累计成本（美元）把成本当作一等状态，而不是只在日志中出现。
totalAPIDuration累计 API 总耗时用于衡量整体请求路径的时间成本。
totalAPIDurationWithoutRetries剔除重试后的 API 耗时可区分“真实工作时长”和“失败恢复开销”。
totalToolDuration工具执行总耗时把模型调用与工具调用分开计量。
totalLinesAdded累计新增行数直接连接到代码变更规模。
totalLinesRemoved累计删除行数与新增行一起构成会话 churn 指标。
lastDuration最近一次请求耗时适合状态栏或最近操作反馈。
modelUsage按模型聚合的使用情况支撑多模型场景下的成本与 token 分布观测。
 
cost-tracker.ts
├── totalCostUSD
├── totalAPIDuration
├── totalAPIDurationWithoutRetries
├── totalToolDuration
├── totalLinesAdded
├── totalLinesRemoved
├── lastDuration
└── modelUsage

为什么要同时记录 API 与 Tool 时间

Claude Code 不是纯聊天应用。它既发起模型请求，也调用本地工具。把 totalAPIDuration 与 totalToolDuration 分开，才能观察瓶颈到底来自模型交互还是工具执行。

为什么记录增删行

totalLinesAdded / totalLinesRemoved 让“会话产出”变得可量化。它们把抽象的 AI 助手活动映射到具体代码 churn，是状态可观测性的另一层。

从架构角度看，cost-tracker.ts 不是单纯统计工具，而是会话状态的一部分：这些字段既可展示给用户，也可在会话恢复、状态栏或统计视图中被重新读取。

可观测性与共享策略：Context、迁移、指标三条线如何互补

如果把 Claude Code 看作一个长期运行的终端应用，那么它的状态策略并不是单一技术选型，而是三条互补的线：

线 1：Context 共享 在 React 树内传播“当前会话正在发生什么”，例如覆盖层、提示、通知、语音与排队消息。
线 2：Migration 演进 在应用启动时整理“历史状态该如何进入当前版本”，避免 UI 直接处理遗留格式。
线 3：Metrics 观测 在运行期间累计“应用到底做了多少事、花了多少时间、花了多少钱”。

架构层面的分工

  • Queue-driven UI/message flow：解决“事件怎么进入系统、怎么有序消费”。
  • Context-based state sharing：解决“消费后的状态如何在组件树里被共享”。
  • Settings evolution via migrations：解决“旧状态如何安全进入新版本”。
  • Metrics/state observability：解决“系统运行后怎样被量化与回放”。
不是一个 store 解决所有问题，而是四个层级各司其职：
1. Queue        -> 流动
2. Context      -> 传播
3. Migration    -> 演进
4. Metrics      -> 观察

这种设计对终端应用尤其合理：终端 UI 更新频繁、异步来源多、会话可能跨多次启动延续、用户又需要看到成本与耗时。Claude Code 把这些问题拆给不同机制处理，因此整体状态模型更清晰，也更容易在未来扩展。

7

端到端数据路径：一次会话如何穿过这些状态层

将上述事实串起来，可以得到一个较稳健的端到端图景：启动阶段先运行迁移，把旧设置收敛到当前结构；运行阶段用户输入、系统事件与任务结果进入 mailbox/queue；React 组件通过多个 Context 读取这些状态并决定如何渲染；同时成本与性能指标持续落入 cost-tracker.ts 与统计层。

① 应用启动
   └─ 11 个 migrations 统一历史设置格式

② 会话运行
   └─ 输入 / 系统事件 / 任务结果进入 mailbox 与消息队列

③ UI 共享
   ├─ QueuedMessage / Notifications 控制消息展示顺序
   ├─ Modal / Overlay / PromptOverlay 控制交互层次
   ├─ Voice 管理语音输入状态
   └─ fpsMetrics / stats 记录可观测性数据

④ 会话记账
   └─ totalCostUSD / totalAPIDuration / totalToolDuration / modelUsage 等字段持续累积

结论

Claude Code 的数据流与状态管理并不是“Redux 式单仓库”，而是一个由队列、上下文、迁移与指标四部分组成的组合架构。它最值得注意的地方在于：把终端 UI 的交互即时性、配置兼容性与会话可观测性放在同一个体系里处理，同时又避免让任何单个机制承担全部责任。
