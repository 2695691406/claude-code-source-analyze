🏗️ 架构总览

QueryEngine 类是整个对话系统的核心入口。每个会话对应一个 QueryEngine 实例，它独占查询生命周期的完整控制权：从消息提交、API 调用、工具调度到结果回流，所有环节均在此类内完成闭环。

class QueryEngine {

🔄 查询管道 (Query Pipeline)

query.ts 是 QueryEngine 的核心管道实现。整个查询流程采用 AsyncGenerator 模式，通过 yield 实现流式事件回传。管道由 query() 入口进入 queryLoop() 递归循环，循环内执行 API 调用 → 工具分派 → 结果回流。

query() 函数签名

export async function* query(
  params: QueryParams
): AsyncGenerator<
  StreamEvent | RequestStartEvent | Message | TombstoneMessage | ToolUseSummaryMessage,
  Terminal // return type — 终止信号
>

返回类型使用 AsyncGenerator 三泛型：Yield 流式事件、Return 终止信号、Next 未使用。这使得调用方可以用 for await...of 消费流式事件，同时通过 .return() 获取终止状态。

管道状态 (State) 核心字段

interface QueryState {
  messages: Message[]; // 消息历史
  toolUseContext: ToolUseContext; // 工具执行上下文
  autoCompactTracking: AutoCompactState; // 自动压缩追踪
  maxOutputTokensRecoveryCount: number; // 输出截断恢复次数
  hasAttemptedReactiveCompact: boolean; // 是否已尝试响应式压缩
  maxOutputTokensOverride: number | null; // 输出 Token 上限覆盖
  pendingToolUseSummary: ToolUseSummary[]; // 待处理工具摘要
  stopHookActive: boolean; // 停止钩子激活
  turnCount: number; // 当前轮次计数
  transition: Transition | null; // 状态转换信号
}

⚙️ 工具执行引擎

StreamingToolExecutor 是工具执行的核心引擎，管理并发控制、AbortController 层级和工具间依赖。它通过 canExecuteTool() 检查工具的 isConcurrencySafe 标志，实现读写分离的并发策略。

并发控制模型

// 并发决策逻辑
function canExecuteTool(tool: Tool): boolean {
  if (tool.isConcurrencySafe()) {
    return true; // ✅ 可并发执行（只读工具）
  }
  return false; // ❌ 串行执行（写入工具）
}
✅ 并发安全：Read, Glob, Grep, LSP ❌ 串行执行：Write, Edit, Bash

🎯 工具编排策略

toolOrchestration.ts 中的 runTools() 函数是工具编排的核心。它将 Claude 返回的工具调用列表按并发安全性分区，然后分别用并行或串行策略执行。

runTools() 编排流程

async function* runTools(
  toolCalls: ToolCall[],
  context: ToolUseContext
): AsyncGenerator<ToolEvent> {

  // Step 1: 分区 — 按并发安全性拆分
  const { concurrent, serial } = partitionToolCalls(toolCalls);

  // Step 2: 并行执行所有并发安全工具
  yield* all(concurrent.map(tc => executeTool(tc)));

  // Step 3: 串行执行非并发安全工具
  for (const tc of serial) {
    yield* executeTool(tc);
  }
}

partitionToolCalls() 分区逻辑

function partitionToolCalls(
  toolCalls: ToolCall[]
): { concurrent: ToolCall[]; serial: ToolCall[] } {
  return {
    concurrent: toolCalls.filter(tc => tc.tool.isConcurrencySafe()),
    serial: toolCalls.filter(tc => !tc.tool.isConcurrencySafe()),
  };
}

并发生成器 — all() 实用函数

all() 是并发执行多个 AsyncGenerator 的核心工具。它同时启动所有生成器，通过 Promise.race 收集第一个完成的结果，合并所有 yield 值，并受并发上限 cap 控制。

Promise.race 驱动 并发上限 cap 合并 yield 值 错误传播

编排策略总览

策略适用工具执行方式上限
并行 Read, Glob, Grep, LSP, WebFetch all() 并发生成器 10 并发
串行 Write, Edit, Bash, Agent for...of 顺序执行 1 串行
混合 同批次含两类工具 先并行后串行 按类型分区

📐 上下文窗口管理

Claude API 有严格的上下文窗口限制。QueryEngine 通过自动压缩追踪（auto-compact tracking）、响应式压缩（reactive compact）和输出截断恢复三重机制，确保长对话不会因为超出窗口而崩溃。

自动压缩追踪 (Auto-Compact Tracking)

// autoCompactTracking 状态字段
interface AutoCompactState {
  totalInputTokens: number; // 累计输入 Token
  compactThreshold: number; // 压缩触发阈值
  lastCompactAt: number; // 上次压缩时的 Token 数
  compactCount: number; // 压缩次数
}

当累计输入 Token 超过阈值时，自动触发 compact 操作——将历史消息压缩为摘要，释放上下文窗口空间。

响应式压缩 (Reactive Compact)

当 API 返回上下文溢出错误时，QueryEngine 会尝试响应式压缩。通过 hasAttemptedReactiveCompact 标志确保每轮最多尝试一次，避免无限压缩循环。

🛑 中断与取消

QueryEngine 的中断机制基于 AbortController 三层层级设计。用户按 ESC 时，信号从根控制器向下传播，逐层取消正在执行的工具。每个工具可以声明自己的 interruptBehavior，决定如何响应中断信号。

AbortController 三层层级详解

层级控制器触发条件影响范围
L1 根 toolUseContext.abortController 用户 ESC · 超时 取消所有工具
L2 兄弟 siblingAbortController Bash 执行错误 取消同批次工具
L3 工具 toolAbortController 单工具超时/错误 仅取消当前工具

信号传播流程

┌──────────────────────────────────────────────────────────┐
│ 用户按 ESC │
│ │ │
│ ▼ │
│ toolUseContext.abortController.abort() │
│ │ │
│ ├──▶ 检查 interruptBehavior │
│ │ ├─ "abort" → 立即取消 │
│ │ ├─ "ignore" → 忽略信号，继续执行 │
│ │ └─ "complete" → 等待当前操作完成后取消 │
│ │ │
│ ├──▶ siblingAbortController.abort() │
│ │ └─ 取消同批次其他工具 │
│ │ │
│ └──▶ toolAbortController.abort() (per-tool) │
│ └─ 取消单个工具执行 │
└──────────────────────────────────────────────────────────┘

interruptBehavior 工具中断行为

abort — 立即中断

收到信号后立即终止执行。适用于：长时间运行的 Bash 命令、WebFetch 等 I/O 密集型工具。

ignore — 忽略信号

完全忽略中断信号。适用于：关键写入操作（如 Write 工具的原子写入），避免半写状态。

complete — 优雅完成

等待当前操作完成后再响应中断。适用于：Edit 工具（等待替换完成），Agent 工具（等待子 Agent 结束）。

🔁 错误恢复机制

withRetry.ts 实现了 5 重错误恢复机制：指数退避重试、529 特殊处理、瞬态容量错误恢复、连接过期重建和自适应延迟。所有 API 调用都经过 withRetry 包装，确保瞬态故障不会导致整个对话中断。

重试核心参数

// withRetry.ts 核心常量
const DEFAULT_MAX_RETRIES = 10; // 默认最大重试次数
const BASE_DELAY_MS = 500; // 基础延迟 500ms
const MAX_529_RETRIES = 3; // 529 状态码最大重试

指数退避策略

// 退避延迟计算公式
delay = BASE_DELAY_MS × 2^retryCount

// 实际延迟序列：
// 重试 0: 500ms
// 重试 1: 1000ms (1s)
// 重试 2: 2000ms (2s)
// 重试 3: 4000ms (4s)
// 重试 4: 8000ms (8s)
// 重试 5: 16000ms (16s)
// ...以此类推到第 10 次

错误分类器

分类器检测条件重试策略上限
is529Error() HTTP 529 过载 立即重试 + 短延迟 3 次
isTransientCapacityError() HTTP 429 / 503 瞬态 指数退避 10 次
isStaleConnectionError() 连接过期 / ECONNRESET 立即重试 10 次
isOverloadedError() 服务端过载 长退避等待 10 次
isContextWindowError() 上下文溢出 触发 reactive compact 1 次

🧰 生成器工具链

generators.ts 提供了一组 AsyncGenerator 工具函数，是整个管道系统的基础设施。这些工具解决了 JavaScript 原生 AsyncGenerator 缺少的并发执行、返回值提取和数组转换等能力。

all() — 并发生成器

async function* all<A>(
  generators: AsyncGenerator<A>[],
  cap?: number // 并发上限，默认 Infinity
): AsyncGenerator<A>

// 用途：同时执行多个 AsyncGenerator，合并所有 yield 值
// 实现：Promise.race + 事件队列
// 场景：并发执行多个工具、并行文件读取

lastX() — 提取最后值

async function lastX<A>(
  generator: AsyncGenerator<A>
): Promise<A | undefined>

// 消费整个生成器，返回最后一个 yield 值
// 用途：当只关心最终结果，不关心中间流式事件

returnValue() — 获取返回值

async function returnValue<Y, R>(
  generator: AsyncGenerator<Y, R>
): Promise<R>

// 消费整个生成器，忽略所有 yield 值，返回 return 值
// 用途：当需要 query() 的终止信号 Terminal，但不消费流式事件

fromArray() — 数组转生成器

async function* fromArray<T>(
  items: T[]

// 将普通数组转换为 AsyncGenerator
// 用途：统一接口，使同步数据可以接入管道系统

设计评析

✅ 巧妙之处

  • AsyncGenerator 管道实现了真正的流式处理——UI 可以实时显示中间结果
  • all() 并发生成器巧妙解决了 JS 缺少并发 iterator 合并的问题
  • 三层 AbortController 实现了精确的中断粒度控制
  • withRetry 错误分类器覆盖了几乎所有瞬态故障场景
  • partitionToolCalls 读写分离确保了并发安全性
  • Reactive compact 让长对话不会因上下文溢出而崩溃

⚠️ 潜在问题

  • 8 层递归管道的调试和错误追踪具有较高复杂度
  • AbortController 三层层级的信号传播可能存在竞态条件
  • 10 次重试 × 指数退避最大等待约 8.5 分钟，用户体验存疑
  • autoCompactTracking 的压缩可能丢失重要的早期上下文
  • isConcurrencySafe 标志为硬编码，缺乏运行时动态评估能力
