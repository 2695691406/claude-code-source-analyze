🧠 三层自愈记忆系统

Claude Code 的记忆并不是单一缓存，而是由持久化文件记忆、查询时相关性召回与会话级摘要记忆组成的三层协作体系。它既强调长期可追溯，又通过入口截断、过时提醒、重复抑制与阈值触发，形成一种“能发现问题、提示修复、避免持续恶化”的架构级自愈能力。

🏗️ 总体架构

从实现上看，Claude Code 的记忆系统可以抽象为三层：第一层是 memdir 管理的持久化文件记忆；第二层是查询时的目录扫描与相关性选择；第三层是 SessionMemory 负责会话内摘要与压缩。三层不是彼此替代，而是各自解决不同时间尺度的问题。

三层结构图

┌──────────────────────────────────────────────────────────────┐
│ Layer 1: 持久化文件记忆 │
│ private/team 目录 + MEMORY.md 入口 + 主题化 *.md 文件 │
├──────────────────────────────────────────────────────────────┤
│ Layer 2: 查询时召回 │
│ scanMemoryFiles() 扫描 frontmatter → findRelevantMemories() 选最多 5 个 │
├──────────────────────────────────────────────────────────────┤
│ Layer 3: SessionMemory │
│ 长对话阈值触发摘要更新，维持会话内压缩记忆与连续性 │
└──────────────────────────────────────────────────────────────┘
长期 查询时 会话级 架构级自愈

为什么说它是“自愈”而不是“自动清理”

代码没有展示一个后台垃圾回收器去定时删除旧记忆；相反，它通过限制入口膨胀、提示过时风险、要求更新或删除错误条目、避免重复写入，把“修复记忆”的责任转交给模型工作流本身。

两种持久作用域

paths.ts 与 teamMemPaths.ts 说明持久化目录分为私有 auto memory 与可选团队 memory。它不是任意层级树，而是围绕“同一项目、不同共享范围”设计的双目录结构。

🧩 四种记忆类型

记忆类型在 memoryTypes.ts 中被硬编码为一个封闭集合，分别是 'user'、'feedback'、'project'、'reference'。这种封闭类型系统非常重要：它迫使系统只保存“当前代码和 git 历史无法直接推导”的上下文。

类型定义（精确事实）

export const MEMORY_TYPES = [
  'user',
  'feedback',
  'project',
  'reference',
] as const

类型职责矩阵

类型 推荐范围 保存内容 典型用途
user 始终私有 用户角色、目标、知识背景、偏好 让解释与协作方式贴合具体用户
feedback 默认私有；项目级规范可升为团队 用户对工作方式的纠正或确认 避免重复犯错，也保留被验证有效的方法
project 私有或团队，偏向团队 目标、截止日期、事件、约束、动机 补足代码仓库外的项目语境
reference 通常团队 外部系统入口与信息指针 告诉模型“去哪里找最新信息”
关键约束：系统明确排除了代码结构、架构、git 历史、调试方案、当前任务细节等“可从项目状态导出”的内容。换言之，记忆系统是补充上下文，不是替代代码检索。

🗂️ 八个核心模块

用户要求覆盖的八个文件，恰好构成持久化记忆层的核心骨架：路径解析、类型定义、入口生成、扫描、相关性选择、安全校验、老化提示与组合提示词生成，各自分工清晰。

模块清单与职责

文件 角色 技术意义
memdir.ts 记忆系统主入口 定义 MEMORY.md、截断规则、提示词拼装与目录存在约束
memoryScan.ts 目录扫描器 递归读取 markdown 头部、抽取 frontmatter、按 mtime 倒序排序、上限 200 文件
findRelevantMemories.ts 相关性选择器 将扫描结果交给 Sonnet 做选择，最多返回 5 个记忆文件
teamMemPaths.ts 团队路径安全层 防遍历、防符号链接逃逸、防 null byte / URL 编码 / Unicode 归一化攻击
memoryAge.ts 陈旧度注释层 把 mtime 变成人类可理解的“几天前”，并在 >1 天时给出风险提示
memoryTypes.ts 类型法典 定义四类记忆、保存准则、不要保存什么、访问记忆时的漂移警告
teamMemPrompts.ts 双目录提示词构建器 在 private + team 模式下生成组合提示，强调双入口索引都要受限
paths.ts auto memory 路径解析器 处理开关、覆盖、配置源优先级、git root 归一化与项目隔离

入口常量

export const ENTRYPOINT_NAME = 'MEMORY.md'
export const MAX_ENTRYPOINT_LINES = 200
export const MAX_ENTRYPOINT_BYTES = 25000

团队模式补充

teamMemPrompts.ts 明确指出 private 与 team 两个目录各自拥有自己的 MEMORY.md，并且两个入口都会被注入上下文，因此两个索引都必须保持简洁。

🛰️ 分层扫描与入口治理

这里的“分层扫描”不是多租户数据库式的层级搜索，而是围绕 项目作用域 → 目录作用域 → 入口索引 → 主题文件 逐层展开。代码中最稳定、最可证实的层次，是 per-project 路径隔离、private/team 双目录，以及目录内的 MEMORY.md + 主题文件结构。

项目与目录层次

~/.claude 或远程 memory base
└─ projects/sanitized-git-root/
   └─ memory/
      ├─ MEMORY.md # private 入口索引
      ├─ *.md # private 主题记忆
      └─ team/
         ├─ MEMORY.md # team 入口索引
         └─ *.md # team 主题记忆

扫描算法

1
定位目录：paths.ts 决定 auto memory 根目录；启用 team memory 时，teamMemPaths.ts 派生团队子目录。
2
递归枚举：scanMemoryFiles(memoryDir) 使用 readdir(..., { recursive: true }) 找到目录下所有 .md 文件。
3
入口排除：扫描时显式排除 MEMORY.md，因为入口索引已在系统提示中直接加载，不需要重复参与相关性候选。
4
头部抽样：每个候选文件只读取前 30 行，用来解析 frontmatter 中的 type 与 description。
5
新鲜度排序：结果按 mtimeMs 倒序排列，只保留最新的 200 个。

入口索引治理：200 行 + 25KB 双闸门

memdir.ts 的 truncateEntrypointContent() 同时限制入口索引的行数与字节数：先按自然换行裁切到 MAX_ENTRYPOINT_LINES = 200，再按 MAX_ENTRYPOINT_BYTES = 25000 做字节级截断，避免长行索引在“行数没超但体积爆炸”的情况下污染上下文窗口。

const wasLineTruncated = lineCount > 200
const wasByteTruncated = byteCount > 25000

// 先切行，再切字节；最终附加 WARNING 提示
自动扫描发现 的真正含义是：系统无需预先维护一张独立数据库表，而是直接把 memory 目录当作事实源，在查询时递归扫描、抽样 frontmatter、排序并构造候选清单。

🎯 相关性匹配

Claude Code 没有在这部分使用向量索引或关键词规则引擎，而是把扫描后的 manifest 交给一个侧路 Sonnet 调用，让模型从文件名、描述、时间戳与最近使用工具中做选择。这种设计把“召回”与“主回答”拆开，降低主线程噪音。

候选清单格式

- [user] user_profile.md (2026-03-05T12:34:56.000Z): senior Go engineer, new to React
- [feedback] testing_policy.md (2026-03-02T08:10:00.000Z): integration tests must hit real DB
- [reference] ingest_linear.md (2026-03-01T06:20:00.000Z): pipeline bugs tracked in Linear INGEST

选择逻辑

  • 过滤 alreadySurfaced，避免重复把同一记忆反复塞回上下文。
  • 把候选 manifest 和用户查询交给 Sonnet 侧查询。
  • 返回严格 JSON：selected_memories: string[]。
  • 只接受真实存在于当前候选集合中的文件名。
  • 最多选择 5 个记忆文件。

工具感知去噪

如果近期已经在使用某个工具，系统会告诉选择器：不要再召回该工具的纯参考说明文档；但若记忆里保存的是该工具的坑点、警告或已知问题，仍然应该召回。这是很典型的“相关 ≠ 有用”过滤。

召回流程图

Query
  ↓
scanMemoryFiles() → 读取 .md 头部 → mtime 倒序 → 最多 200
  ↓

  ↓

注意边界：这里的相关性匹配确实是“模型驱动的选择”，但并没有证据表明它维护任何长期 embedding 索引、相似度缓存或跨项目共享召回池。报告中应避免把它描述成向量数据库。

🧮 SessionMemory 阈值

持久化文件记忆之外，Claude Code 还有一层会话级摘要记忆。它主要针对长对话压缩和跨轮连续性，而不是替代 memdir 主题文件。是否触发更新，取决于三个精确阈值。

默认阈值（精确事实）

export const DEFAULT_SESSION_MEMORY_CONFIG = {
  minimumMessageTokensToInit: 10000,
  minimumTokensBetweenUpdate: 5000,
  toolCallsBetweenUpdates: 3,
}

阈值含义

配置项 值 控制点 含义
minimumMessageTokensToInit 10000 初始化门槛 上下文窗口累计到 10K token 后，SessionMemory 才被正式启用
minimumTokensBetweenUpdate 5000 更新门槛 距离上次提取后，至少再增长 5K token 才允许下一轮提取
toolCallsBetweenUpdates 3 活跃度门槛 工具调用数达到 3 次时，配合 token 条件决定是否抽取摘要

更新判定要点

const shouldExtract =
  (hasMetTokenThreshold && hasMetToolCallThreshold) ||
  (hasMetTokenThreshold && !hasToolCallsInLastTurn)
  • Token 阈值始终是硬前提；工具调用阈值不能单独触发更新。
  • 若最后一轮没有工具调用，也可在“自然停顿点”触发提取。
  • 这使 SessionMemory 更像“按对话节奏抽取摘要”，而不是每轮都刷新。

故障恢复阈值

除了更新门槛，sessionMemoryUtils.ts 还定义了 EXTRACTION_WAIT_TIMEOUT_MS = 15000 与 EXTRACTION_STALE_THRESHOLD_MS = 60000。若某次提取卡住太久，后续流程不会无限等待。

写权限最小化

SessionMemory 的 forked agent 只被允许对精确的 session memory 文件执行 Edit。这说明该层既追求持续压缩，也强调“只改允许改的那一份摘要”。

♻️ 自愈、陈旧控制与剪枝边界

“自愈”必须谨慎表述。基于代码，我们可以确认系统拥有多种纠偏机制；但同样必须承认：没有看到一个自动 TTL 删除器或后台 prune daemon。更准确的说法是，它实现了面向模型工作流的自愈协议。

可被代码直接支持的自愈机制

  • 入口防膨胀：MEMORY.md 同时受 200 行与 25KB 双上限约束，溢出时附带警告。
  • 重复抑制：提示词明确要求先检查是否已有可更新的记忆，避免重复文件和重复条目。
  • 错误修复：提示词明确写出“若记忆被证明错误或过时，应更新或移除”。
  • 陈旧提示：memoryAge.ts 在记忆超过 1 天时注入“点时间观察，不是实时状态”的新鲜度告警。
  • 提取超时自恢复：SessionMemory 卡住超过 60 秒会被视为 stale，不再阻塞后续流程。

陈旧度提示不是删除器

if (d <= 1) return ''
return "This memory is N days old... Verify against current code before asserting as fact."

这里的设计哲学非常清楚：系统不是直接替用户删掉“老文件”，而是通过提示让模型意识到该记忆可能已经漂移，需要去当前代码或外部资源重新核验。

可以安全推断的“剪枝”

  • 入口索引被截断，相当于对上下文注入做硬剪枝。
  • 扫描候选只保留最新 200 文件，是召回集合级剪枝。
  • 相关性选择最多 5 个，是最终注入级剪枝。

不能过度宣称的内容

  • 没有证据显示存在按年龄自动删除旧记忆的后台任务。
  • 没有看到基于分数的全局 LRU / LFU 淘汰器。
  • 没有看到数据库级 compaction；更多是提示词驱动的人工整理。
报告写法建议：可以说这是“自愈式记忆系统”，因为它会检测、警告、限制并引导修正；但不应写成“全自动后台清理系统”，那会超出代码能证明的范围。

🔄 端到端工作流

最终把上述实现串起来，可以看到 Claude Code 的记忆系统并不是一次性加载全部记忆，而是依据不同时间尺度进行渐进式治理：入口索引负责“总览”，扫描与选择负责“按需召回”，SessionMemory 负责“长对话压缩”。

完整工作链

1
路径建立：paths.ts 根据环境变量、设置与 canonical git root 解析 auto memory 目录；team 模式下再派生团队目录。
2
系统提示注入：memdir.ts / teamMemPrompts.ts 把类型规则、保存规则、访问规则与 MEMORY.md 入口内容送入上下文。
3
查询时扫描：memoryScan.ts 递归扫描主题文件，只读头部，按修改时间倒序构造 manifest。
4
相关性召回：findRelevantMemories.ts 让 Sonnet 从 manifest 中选出最多 5 个明确有帮助的记忆。
5
新鲜度提醒：读取具体记忆时，memoryAge.ts 可把 mtime 解释成“today / yesterday / N days ago”，并在旧记忆上附加 verify 提醒。
6
会话摘要补位：当对话足够长时，SessionMemory 依据 10000 / 5000 / 3 的阈值抽取摘要，支撑长会话持续运行。

结论：为什么这个设计有效

它避免了“全量装载所有历史记忆”的上下文浪费，也避免了“完全无状态”的重复解释成本。更重要的是，系统把长期记忆、即时相关性和会话压缩拆成三层，每一层都有自己的容量约束和纠偏策略，因此整体更稳健。
