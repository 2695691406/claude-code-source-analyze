1 🔄 AsyncGenerator 流式查询引擎 QueryEngine.ts (1,295 LOC) + query.ts (1,729 LOC)

Claude Code 最核心的创新在于将 AsyncGenerator 作为所有层间通信的统一协议。从用户输入到 API 调用、工具执行、消息流转，全部基于异步生成器管道。

// 三层嵌套的 AsyncGenerator 管道
async *submitMessage(prompt): AsyncGenerator<SDKMessage>  // QueryEngine 层
  └── async function* query(params): AsyncGenerator<Message>  // 查询循环层
        └── async function* runAgent(config): AsyncGenerator<Message>  // Agent 执行层
  • ✦生产者-消费者解耦：上层消费者按需拉取，下层生产者按需生成
  • ✦原生背压支持：消费速度慢时自动暂停生产，无需额外缓冲区管理
  • ✦优雅取消：通过 AbortController 传播取消信号，自动清理资源
  • ✦状态交织：在响应流中穿插进度更新、权限请求、工具调用结果
  • ✦类型安全：TypeScript 泛型确保消息类型在管道中传播正确
  • ✦七阶段生命周期：构造 → 初始化 → 本地快捷路径 → 文件历史快照 → 查询循环 → 结果验证 → 清理
🏭 产品化方向：企业级流式查询引擎 SDK
适用场景：任何需要流式 AI 对话的企业应用
核心价值：开箱即用的流式管道、背压控制、优雅取消
差异化：相比 LangChain 的 Callback 模式，AsyncGenerator 更优雅、类型安全
2 🔐 四层权限控制系统 permissions/ + query.ts + Tool.ts

业界首个专为 AI Agent 设计的四层权限模型，从编译时到运行时交互层层把控。

第一层：工具池组装（编译时 + 启动时）
  ├── Feature Gate 编译时裁剪
  ├── 模式过滤（如 Plan 模式禁用写入工具）
  └── Agent 类型过滤（子 Agent 可限制工具集）

第二层：权限模式快速路径（运行时全局）
  ├── bypass → 全部允许 │ dontAsk → 全部拒绝
  └── acceptEdits → 文件编辑允许 │ default → 进入规则匹配

第三层：规则引擎匹配（运行时按规则）
  策略规则(policy) → 用户规则 → 项目规则 → 本地规则 → 标记规则 → 会话规则

第四层：用户审批（运行时交互）
  ├── AI 分类器判断 │ PreToolUse Hooks │ 用户确认对话框
  • ✦Bash 安全分类器（23+ 检测器）：命令注入、变量注入、重定向、编码欺骗检测
  • ✦两阶段设计：静态正则（快速）→ AI 分类器（精确）
  • ✦竞速优化：分类器与 UI 并行运行，结果到达前可跳过提示
  • ✦决策可追溯性：每个权限决策都记录来源，支持事后审计
🔐 产品化方向：企业级 Agent 访问控制框架
适用场景：任何多工具 AI Agent 系统的安全管控
核心价值：细粒度权限控制、规则 DSL、AI 安全评估、审计追溯
差异化：业界首个专为 AI Agent 设计的四层权限模型
3 🧠 五层上下文压缩管道 services/compact/ (1,500+ LOC)

业界最精细的五层压缩管道，解决 AI 模型上下文窗口有限但长对话快速耗尽窗口的问题。

第一层：工具结果预算（Tool Result Budget）→ 单次工具输出超限时截断
第二层：历史裁剪（History Snip）→ 旧对话历史压缩为摘要
第三层：微压缩（Microcompact）→ 基于时间窗口的逐消息压缩
第四层：上下文折叠（Context Collapse）→ 读取时投影视图，归档旧内容
第五层：自动压缩（Auto Compaction）→ AI 驱动完整对话重写 + 熔断机制
  • ✦缓存感知压缩（Cache-Aware）：避免修改缓存前缀，确保 Prompt Cache 不失效
  • ✦Fork 子 Agent 共享完全相同的前缀 → 只有一次缓存未命中
  • ✦压缩后恢复：重新注入 Top 5 文件内容（50KB）+ Skills（25KB）
  • ✦熔断机制：连续 3 次失败后停止
🧠 产品化方向：AI 会话上下文智能管理器
适用场景：所有长对话 AI 应用（客服、编程助手、数据分析）
核心价值：自动化上下文管理、缓存友好、成本优化
差异化：五层递进压缩 + 缓存感知，业界独创
4 ⚡ 推测性执行引擎 tasks/ + query.ts

当用户阅读 AI 响应时，系统 Fork 一个 Agent 预测下一轮操作并预先执行，可达 40% 感知延迟降低。

用户输入 → AI 响应 → 用户阅读（空闲时间）
                         ↓
              Fork Agent 预测下一步 → 写入 Overlay 文件系统
                         ↓
              用户输入匹配预测 → 合并 Overlay（零拷贝）
              用户输入不匹配 → 丢弃 Overlay
  • ✦边界检测：Bash 命令 → 停止推测（副作用不可逆）
  • ✦Edit/Write 工具 → 停止推测（文件修改需确认）
  • ✦权限拒绝 / 自然完成 → 停止推测
⚡ 产品化方向：AI 交互预测加速器
适用场景：AI 编码助手、智能客服、数据分析
核心价值：显著降低感知延迟，提升交互流畅度
差异化：Overlay 文件系统 + 安全边界检测
5 🎯 Feature Gate 编译系统 Bun Macro 编译时代码消除

基于 Bun 宏在编译时将 feature() 求值为常量布尔值，Terser 随后消除不可达分支，实现零运行时开销的功能开关。

// Bun 宏编译时求值
if (feature('KAIROS')) {
  // gate 为 false 则完全移除 — 甚至字符串字面量也不泄漏
}

20+ 已知 Gate: KAIROS, COORDINATOR_MODE, BRIDGE_MODE,
  VOICE_MODE, DAEMON, WORKFLOW_SCRIPTS, BUDDY, FORK_SUBAGENT,
  UDS_INBOX, TORCH, CONTEXT_COLLAPSE, TERMINAL_PANEL ...
  • ✦零性能开销：死代码完全移除（vs 运行时 Flag 每次检查有微小开销）
  • ✦零代码泄漏：字符串也移除（vs 运行时全部包含在产物中）
  • ✦适用场景：产品线分离（vs 运行时适用 A/B 测试、灰度发布）
🎯 产品化方向：多版本产品编译工具链

核心价值：零运行时开销、零代码泄漏、类型安全
差异化：编译时消除 vs 运行时判断
