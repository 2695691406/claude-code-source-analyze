1 AsyncGenerator 流式管道模式 统一通信协议

问题：AI Agent 需要处理流式响应、支持取消、管理背压，传统的回调或 Promise 模式难以优雅处理。

💡 方案
使用 AsyncGenerator 作为所有层间通信的统一协议 
User Layer          Query Layer           API Layer
    │                   │                    │
    │  for await ──→    │  for await ──→     │
    │  (messages)       │  (responses)       │  API Stream
    │                   │                    │
    │  ◄── yield        │  ◄── yield         │
    │     message       │     response       │
    │                   │                    │
    │  abort ──────→    │  abort ──────→     │
    │  (cancel)         │  (cancel)          │  Cancel
// 核心模式：三层嵌套的 AsyncGenerator
async function* submitMessage(prompt: string): AsyncGenerator<Message> {
  const context = prepareContext(prompt);
  yield* queryLoop(context);    // yield* 优雅传播
  await cleanup();
}

async function* queryLoop(context: QueryContext): AsyncGenerator<Message> {
  while (true) {
    for await (const chunk of apiStream(context)) {
      if (chunk.type === 'tool_use') {
        yield* executeToolAndYield(chunk);
      } else {
        yield chunk;   // 直接传播给上层
      }
    }
    if (shouldStop(context)) break;
  }
}
  • ✦背压控制：原生支持，消费者按需拉取，无需额外缓冲区管理
  • ✦取消支持：通过 AbortController + return() 传播取消信号
  • ✦嵌套组合：yield* 优雅组合，避免回调地狱
  • ✦类型安全：TypeScript 泛型确保消息类型在管道中正确传播
  • ✦懒求值：按需生成，不会提前计算不需要的结果
  • ✦调试友好：完整的栈追踪，优于回调/Observable 方案
AI 对话流实时数据管道多步骤工作流流式 API 网关
2 QueryEngine 会话引擎模式 每 Agent 一个实例

问题：AI Agent 需要管理会话状态、支持多种执行模式，且每个 Agent 实例需要独立的状态空间。

💡 方案
每个 Agent = 一个 QueryEngine 实例，通过配置注入而非继承来支持多种模式 
QueryEngine 实例（每 Agent 一个）
├── 不可变配置
│   ├── model: 模型选择
│   ├── tools: 工具集（经过权限过滤）
│   ├── systemPrompt: 系统 Prompt
│   ├── maxTurns: 最大轮次
│   └── canUseTool: 权限判断函数
├── 可变状态
│   ├── messages: 消息历史
│   ├── fileCache: 文件缓存（COW）
│   ├── totalUsage: Token 使用量
│   └── permissionDenials: 权限拒绝记录
└── 生命周期
    ├── submitMessage(): 七阶段处理
    ├── abort(): 取消当前查询
    └── dispose(): 释放资源
七阶段处理生命周期：
1. Construct    → 创建 AbortController、初始化文件缓存
2. Initialize   → 权限包装、系统 Prompt 组装
3. Shortcut     → 斜杠命令短路（跳过 API）
4. Snapshot     → Git 快照（记录文件状态）
5. Query Loop   → 核心消息泵（API 调用 + 工具执行循环）
6. Validate     → 结果校验（结构化输出、成功条件）
7. Cleanup      → 使用量统计、记忆提取、会话刷新
  • ✦配置注入而非继承：不需要 CLIQueryEngine、SDKQueryEngine 等子类
  • ✦COW 文件缓存：Agent Fork 时共享缓存，修改时拷贝
  • ✦七阶段明确分离：每个阶段职责清晰，便于扩展和调试
多 Agent 会话管理多执行模式 AI 应用精细状态对话系统
3 工具抽象与注册模式 统一 Tool 接口 + Zod 校验

问题：Agent 需要调用多种工具（文件操作、API 调用、Shell 执行等），需要统一的抽象来管理工具的发现、校验、执行和权限。

💡 方案
统一的 Tool 接口 + Zod Schema 校验 + 动态注册 
// 统一工具接口
interface Tool<TInput, TOutput> {
  name: string;
  description: string | (() => string);   // 支持动态描述
  inputSchema: ZodSchema<TInput>;         // Zod 运行时校验
  call(input: TInput, context: ToolContext): Promise<TOutput>;
  checkPermissions?(input: TInput): PermissionResult;
  isConcurrencySafe: boolean;             // 默认 false（fail-closed）
  isEnabled?(context: AppState): boolean;
  userFacingName?(): string;
}
工具集聚合
├── 内置工具（编译时注册，40+）
├── MCP 工具（运行时发现，动态 Schema 翻译）
├── Skill 工具（运行时加载，零代码定义）
├── Plugin 工具（进程内，深度集成）
└── Feature Gate 工具（编译时裁剪）

流式工具执行器（StreamingToolExecutor）
├── 工具调用入队（流式收集参数）
├── 权限预检（与 API 流并行）
├── 并发安全工具 → 并行执行
├── 非并发安全工具 → 串行执行
└── 结果收集 → ToolResultBlock
  • ✦Fail-Closed 默认：新工具默认非并发安全（串行），显式声明安全才并行
  • ✦Zod 运行时校验：防止 AI 生成的错误参数到达工具逻辑
  • ✦动态描述：工具描述可根据上下文变化（如当前文件类型）
多工具 Agent插件化架构自动化系统
4 层次化权限决策链模式 七层从粗到细

问题：AI Agent 操作涉及安全敏感操作（文件修改、命令执行、数据访问），需要灵活且安全的权限控制。

💡 方案
七层权限决策管道，从粗到细、从快到慢 
工具调用请求
  │
  ├─ Layer 0: 工具池过滤（该工具是否可用？）
  ├─ Layer 1: 输入校验（参数是否合法？）
  ├─ Layer 2: 权限模式快速路径
  │   ├── bypass → 允许 │ dontAsk → 拒绝
  ├─ Layer 3: 工具级权限检查
  │   └── tool.checkPermissions(input)
  ├─ Layer 4: 规则引擎匹配
  │   ├── policy（组织策略）→ 最高优先级
  │   ├── user / project / local / flag
  │   └── session → 最低优先级
  ├─ Layer 5: PreToolUse Hooks（外部脚本）
  ├─ Layer 6: AI 安全分类器（Bash 两阶段竞速）
  └─ Layer 7: 用户交互确认
// 规则 DSL 设计
interface PermissionRule {
  tool: string;            // 工具名称（支持通配符）
  action: 'allow' | 'deny' | 'ask';
  pattern?: string;        // 参数匹配模式
  scope: 'policy' | 'user' | 'project' | 'local' | 'flag' | 'session';
  priority: number;
  metadata?: { reason: string; expiry?: Date; };
}
  • ✦决策可追溯：每个权限决策都记录来源，支持事后审计
  • ✦两阶段检测：静态正则（快速）→ AI 分类器（精确）
  • ✦竞速优化：分类器与 UI 并行运行

5 递归嵌套 Agent 模式 无限嵌套 + 预算控制

问题：复杂任务需要分解为子任务，由专业化的子 Agent 处理，且子任务可能进一步分解。

💡 方案
支持无限嵌套的 Agent 递归模型，通过预算和工具池控制安全边界 
Main Agent（用户交互）
  ├── Research Agent（信息收集）
  │   ├── CodeSearch Agent（代码搜索）
  │   └── DocSearch Agent（文档搜索）
  ├── Implementation Agent（实现）
  │   ├── Frontend Agent（前端实现）
  │   └── Backend Agent（后端实现）
  │       └── DB Agent（数据库变更）
  └── Review Agent（审查）
      ├── Security Agent（安全审查）
      └── Performance Agent（性能审查）
资源预算：
├── Token Budget：子 Agent ≤ 父 Agent 剩余预算
├── USD Budget：全局美元预算上限
├── Max Turns：每个 Agent 的最大轮次数
└── Time Budget：超时自动终止

工具池控制：
├── 继承模式：子 Agent 继承父 Agent 的工具子集
├── 排除模式：可排除 AgentTool 防止无限递归
├── 限制模式：轻量级 Agent 只有只读工具
└── 定制模式：YAML 定义精确的工具白名单

Fork 优化：
├── 父消息历史共享前缀 → Prompt Cache 命中
└── 所有 Fork 子 Agent 共享缓存 → 单次 Cache Miss
  • ✦权限只严不松：子 Agent 权限 ≤ 父 Agent 权限
  • ✦dontAsk 模式：防止子 Agent 弹出交互提示
  • ✦COW 文件缓存：Fork 时共享，修改时拷贝
复杂任务分解多专家协作递归搜索分析

7 上下文窗口智能管理模式 五层递进压缩 + 缓存感知

问题：AI 模型上下文窗口有限，长对话和大量工具结果会耗尽窗口，但简单截断会丢失关键信息。

💡 方案
五层递进压缩 + 缓存感知策略 
原始上下文 [500K tokens]
  │
  ├─ Layer 1: 工具结果预算
  │  └── 单次工具输出超 50K → 截断至摘要        → [350K]
  │
  ├─ Layer 2: 历史裁剪
  │  └── 老对话 → AI 生成摘要替代                → [200K]
  │
  ├─ Layer 3: 微压缩（按工具类型独立策略）
  │  │   FILE_READ: 保留文件名 + 关键行
  │  │   Bash: 保留命令 + 退出码 + 最后 N 行
  │  │   Search: 保留 Top K 结果
  │  └── 缓存感知：避免修改缓存前缀！            → [120K]
  │
  ├─ Layer 4: 上下文折叠
  │  └── 归档旧内容，读取时投影                  → [100K]
  │
  └─ Layer 5: 自动压缩（最后手段）
     └── AI 驱动完整对话重写 + 熔断机制           → [80K]
缓存感知的关键意义：
├── API 缓存前 N 个 Token（系统 Prompt + 历史前缀）
├── 后续请求前缀不变 → Cache Hit（便宜 90%）
├── 前缀变了 → Cache Miss（全价重算）
└── 因此压缩时必须保护前缀不变！

微压缩策略：
├── 只压缩缓存窗口之后的内容
├── Fork 子 Agent 共享相同前缀 → 单次 Miss
└── snipTokensFreed 精确传递释放量
  • ✦缓存感知压缩：避免修改缓存前缀，确保 Prompt Cache 不失效
  • ✦熔断机制：连续 3 次自动压缩失败后停止尝试
  • ✦压缩后恢复：重新注入 Top 5 文件 + Skills 内容
长对话 AI大上下文分析成本优化
8 Agent 记忆与学习模式 三层记忆 + Dream 巩固

问题：AI Agent 每次对话都从零开始，无法积累经验和知识，团队知识无法在 Agent 间共享。

💡 方案
三层记忆架构 + Dream 巩固机制 + 团队同步 
长期记忆（跨会话持久化）
├── 用户级 ~/.claude/agent-memory/
├── 项目级 .claude/agent-memory/
└── 本地级 .claude/local-agent-memory/

中期记忆（项目级配置）
├── CLAUDE.md（根目录） → 项目规范、编码约定
├── 嵌套 CLAUDE.md      → 模块特定约定
└── 团队记忆（远程同步） → 团队共享知识

短期记忆（会话内）
├── 消息历史、文件缓存（COW）、权限决策缓存
Dream 巩固机制：
会话结束 → DreamTask 触发
├─ Gate 1: 时间间隔检查（距上次 Dream > N 小时？）
├─ Gate 2: 会话数量检查（累积会话 > N 次？）
├─ Gate 3: 文件锁检查（无其他 Dream 在运行？）
└─ 通过所有 Gate：
   ├── Fork 子 Agent 分析会话 → 提取洞察 → 写入长期记忆
   └── 冲突检测 + 后台静默完成

团队记忆同步：
├── Pull: 服务端优先 │ Push: 内容哈希增量同步
├── 安全: 秘密扫描（上传前检测凭证）
└── 约束: 250KB 上限 / 本地删除不传播
  • ✦三层分离：长期/中期/短期记忆各司其职
  • ✦Dream 后台巩固：不影响用户交互的知识提取
  • ✦安全上传：秘密扫描防止凭证泄露
跨会话学习团队知识共享持久化 Agent
9 推测性执行模式 预测 → 匹配 → 合并

问题：AI Agent 的响应存在延迟，用户等待时间影响体验。

💡 方案
在用户空闲时 Fork Agent 预测下一步操作，匹配则合并，不匹配则丢弃 
主线程                     推测线程
  │                          │
  ├─ AI 响应完成              │
  │                          │
  ├─ 用户阅读中... ──────→   Fork Agent 启动
  │                          ├─ 预测下一步操作
  │                          ├─ 执行操作
  │                          └─ 写入 Overlay FS
  │                          
  ├─ 用户输入 ─────────→    匹配检查
  │                          ├─ 匹配 → 合并 Overlay（零拷贝）
  │                          └─ 不匹配 → 丢弃 Overlay
  │
  └─ 继续处理

安全边界（自动停止）：
├── Bash 命令（副作用不可逆）
├── Edit/Write（需确认）
├── 权限拒绝 │ 网络请求
└── 自然完成
  • ✦可预测工作流：~40% 感知延迟降低
  • ✦不可预测工作流：~0% 改善（仅消耗少量计算）
  • ✦平均效果：~20% 感知延迟降低

10 配置即策略模式 Config as Strategy

问题：Agent 系统需要支持多种行为模式（调试、安全、快速等），传统的继承或策略模式导致代码爆炸。

💡 方案
一个核心引擎 + 配置对象决定全部行为，无需子类或策略接口 
// 传统方式：N 种模式 = N 个子类
class CLIAgent extends BaseAgent { ... }
class SDKAgent extends BaseAgent { ... }
// 模式组合爆炸：CLI+Debug, SDK+Safe...

// Claude Code 方式：一个配置对象
interface AgentConfig {
  model: ModelConfig;
  tools: Tool[];
  canUseTool: PermissionChecker;
  maxTurns: number;
  bare: boolean;
  getAppState: () => AppState;
  setAppState: (state: AppState) => void;
  onMessage: MessageHandler;
  systemPrompt: string;
  // ... 20+ 配置项
}
const engine = new QueryEngine(config); // 同一个引擎
维度继承层次策略模式配置即策略
代码重复高（方法覆盖）中（接口实现）低（配置变化）
组合爆炸高（NxM子类）中（NxM策略）低（配置组合）
运行时灵活低（编译时绑定）中（构造时注入）高（随时修改）
理解难度高（继承链）中（多接口）低（单配置对象）

10 大设计模式构成完整的 Agent 系统设计方法论

Claude Code 的 10 大设计模式构成了一个完整的 Agent 系统设计方法论。这些模式不是孤立的，而是相互支撑、协同工作的。掌握这套方法论，就能设计出安全、高效、可扩展的 AI Agent 系统。
